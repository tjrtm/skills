# GenAI App Security Audit

A working security auditor for codebases built with AI assistance. Bundles a 258-rule checklist with concrete grep/AST/runtime detection methods. Adapted from [`a-leks/genai-app-security-checklist`](https://github.com/a-leks/genai-app-security-checklist) (Apache 2.0).

The checklist covers traditional web app vulnerabilities **plus** a dedicated 33-item category for LLM integration issues — prompt injection, indirect prompt injection, vector-DB tenant isolation, MCP tool poisoning, agentic shell access without sandboxing, cross-agent injection, and similar patterns that show up repeatedly in AI-generated code.

---

## When to invoke this

Run this skill any time the user wants security feedback on actual code or configuration — whether they frame it as "audit", "review", "is this safe", "find bugs before I ship", or "did the AI leave any holes". The skill works for full-repo sweeps as well as narrow checks of a single file, a single category, or a single concern (e.g., "just prompt injection").

Do **not** invoke for purely educational questions ("what is SSRF") or for fixing a single bug the user has already identified ("my JWT verify is failing"). Those are direct answers, not audits.

## Before you start

Confirm two things briefly:

1. **Scope.** Whole repo, a directory, specific files, or a category? Default to whole repo if the user has clearly handed you one.
2. **Authorization.** The checklist is for code the user owns or has permission to audit. If the request looks like reconnaissance against a third party ("scan example.com's source for vulnerabilities"), decline.

If you can already see the codebase laid out (uploaded files, attached repo, or you are running in a directory), skip asking and start scanning. Mention scope in your first line of output instead.

## How to run a scan

Both reference files live in `references/` and are loaded on demand:

- `references/scan-checklist.md` — the executable 258-rule detection list with concrete grep patterns, AST queries, and config checks. **Load this when you are actually running checks.** It is your operational playbook.
- `references/vulnerabilities.md` — the summary table with severity ratings (🔴🟠🟡🟢) and detection-type tags ([S] static / [R] runtime / [C] config). **Load this for severity lookups, category breakdowns, or when explaining a finding to the user.**

### Detection method tags

- **[S] Static** — code search (ripgrep, AST queries, file reading). You can do this directly.
- **[R] Runtime** — requires hitting a running server (curl, request to staging). Note these as "requires runtime check" in the report; do not invent results.
- **[C] Config** — environment, provider dashboard, or infrastructure-level. Often unverifiable from source alone; note as "requires manual config check" with what to look for.

You can verify [S] checks fully. [R] and [C] checks you can flag based on indirect evidence (e.g., no `helmet` import suggests headers may not be set, but the only way to know for sure is `curl -I`).

### Workflow

1. **Inventory the stack first.** Spend the first minute identifying language, framework, ORM, auth library, LLM provider(s), DB. This determines which rules apply. Look at `package.json` / `requirements.txt` / `composer.json` / `go.mod`, the entry point, the routes directory. A Next.js + Prisma + OpenAI app has different attack surface than a Laravel + MySQL + no-LLM app — running every rule blind wastes effort. Skip categories with no applicable code (e.g., category 6 if there is no LLM integration; category 10 Stripe rules if there is no Stripe).

2. **Use ripgrep over manual file reads.** For each rule, the detection method tells you what to search for. Run those searches as bash commands. `rg` is faster, recursive by default, and handles ignore files. Batch related rules into one rg invocation when patterns overlap (e.g., 5.1 + 5.2 + 5.9 + 5.10 are all secret-pattern scans).

3. **Open the references on demand.** You do not need to memorize all 258 rules. Open `references/scan-checklist.md`, work through the categories that apply, run the searches, write findings as you go.

4. **For every rule that applies to this stack, produce one explicit result.** PASS / FINDING / N/A. Do not silently skip. Vague summaries like "looks fine on auth" defeat the point of the checklist — the user wants a row per rule.

5. **Findings need actual code, not generic advice.** When you find something, show the offending snippet with file path and line number, then provide the literal replacement code. "Use parameterized queries" is not a remediation; `db.query('SELECT * FROM users WHERE id = $1', [userId])` is.

6. **Distinguish what you verified from what you flagged on indirect evidence.** "JWT secret is the literal string `'changeme'` in `src/auth.ts:12`" is a confirmed [S] finding. "No `helmet()` middleware found in entry point — security headers likely missing" is an inferred [R] finding that needs `curl -I` to confirm. Both are useful; calling them out distinctly preserves the user's trust.

## Scoping shortcuts

When the user narrows the request, respect the narrowing:

- **"Just prompt injection"** → run category 6 only, mainly rules 6.1, 6.2, 6.3, 6.4, 6.5, 6.6, 6.26, 6.27, 6.30. Skip the rest of the checklist.
- **"Check auth"** → category 1 + 2.
- **"Production readiness"** → 5 (secrets), 8 (infra), 9 (frontend), 15 (supply chain). Skip business logic and emerging.
- **"LLM integration only"** → entire category 6.
- **Single file scan** → walk every category but only against that file; many rules will be N/A, that is expected.

## Notes on the checklist itself

- **Category 17 (Emerging Patterns)** is provisional — flag findings but mark them as "requires manual verification" and do not assign hard severity.
- **AI-specific rules (category 6)** are the highest-value section because they catch issues that pre-LLM security tools miss entirely. Prioritize these when the user is auditing code that includes LLM API calls, agentic patterns, RAG, or MCP servers.
- **The checklist is opinionated.** If you disagree with a rule's severity for a specific context, note it in the finding rather than silently downgrading. The user can decide.

## What this skill is not

This is not a replacement for runtime security testing (DAST), professional penetration testing, or formal threat modeling. It catches the most common patterns AI coding assistants produce — which is a high-value first pass, but it is a floor, not a ceiling. Say so in the report's closing line if appropriate.
